# Slidecraft — 良いスライドの原則に基づくデッキ作成スキル

特定ブランドのテンプレートに縛られず、**「スライドはこうあるべき」という原則**を core に据えて、
ストーリー設計 → 品質レビュー → PPTX生成（PptxGenJS）を一気通貫で行う。

- **原則はブランド非依存**（[references/principles.md](references/principles.md)）。
- **配色・フォント・ロゴはテーマトークンで差し替え**（[assets/theme.template.js](assets/theme.template.js)）。ブランドに合わせる。
- **レイアウトは原則の範囲で裁量**を持つ。固定テンプレに当てはめず、メッセージに最適な型を選ぶ。
- **生成は PptxGenJS**（[references/layout-patterns.md](references/layout-patterns.md)）。

## 前提

このスキルは公開 pptx スキルの上に構築される。生成に入る前に `/mnt/skills/public/pptx/SKILL.md`（無ければ pptxgenjs の基本）を理解しておくこと。Node + `pptxgenjs`、PDF確認用に `soffice`(LibreOffice) を使う。

---

## 処理フロー（設計→レビュー→生成）

```
STEP 0: ブランド/テーマ確定 … 配色・フォント・ロゴ・機密ラベルを theme に落とす
STEP 1: インプット収集 … 目的・聞き手・MTGとの関係・「何にYESしてほしいか」
STEP 2: ストーリー設計 … 構成表を出して【ユーザー確認】
STEP 3: 品質レビュー … 各スライドを原則で採点（quality-checklist.md）
STEP 4: 生成 … layout-patterns.md の theme 駆動ヘルパーで PptxGenJS 生成 → PDFで目視確認
```

**STEP 2 の構成表は必ずユーザー確認を取ってから STEP 3・4 に進む**（手戻りコスト最小化）。

---

## STEP 0: ブランド / テーマ確定

`assets/theme.template.js` をコピーして案件用 `theme.js` を作る。**色はブランドに合わせる**のが原則（コーポレートカラー or 指定があればそれ）。指定が無ければ中立デフォルトのまま進め、後で差し替え可能にしておく。

最低限決めるトークン：`primary / primaryDark / navy / accent / text / panelBg / border`、`fonts.header / body`、`logo`(任意)、`confidentialLabel`(任意)。

---

## STEP 1: インプット収集

必ず確認する3点（曖昧なら聞く。答えが構成を変える）：
1. **この資料で何にYESしてもらいたいか**（MTG後の意思決定）
2. **聞き手の役職レベル**（現場 / 部長 / 役員）
3. **前回との関係**（初回 / 継続 / 報告後）

受け付けるパターン：A 新規作成（目的＋メモ）/ B リメイク（既存pptx・pdf）/ C ブラッシュアップ（既存＋観点）。

---

## STEP 2: ストーリー設計（構成表 → ユーザー確認）

「目的→結論→根拠」が一直線に通るアークを設計する。型の引き出しは principles.md の「ストーリーの組み立て」を参照（初回ディスカッション/提案/進捗報告/本格導入 など）。

構成表フォーマット：

```
## 構成案
対象 / MTGタイプ / YESをもらいたいこと / 聞き手
| # | スライドタイトル | 種別 | メッセージ（1行＝そのスライドの結論） |
ストーリーチェック: 冒頭の目的と末尾のNAが呼応 / 論理が飛んでいない / なぜ自社か / Appendixに逃がす詳細が本編に侵入していないか
```

→ ここで必ずユーザー確認。

---

## STEP 3: 品質レビュー

各スライドを [references/quality-checklist.md](references/quality-checklist.md) の5観点で採点する：
**1スライド1メッセージ / タイトル2層（テーマ＋結論ライン）/ So-What（事実→だから何）/ 特徴→便益 / 視認性**。
不合格は分割・Appendix退避・メッセージ追記で直す。

---

## STEP 4: 生成（PptxGenJS）

[references/layout-patterns.md](references/layout-patterns.md) の **theme 駆動ヘルパー** を使う。要点：
- ヘルパーは色を `theme` から引く。**色コードを直書きしない**。
- **ページ番号は自動採番**（カウンタ `pg()`）。区切り・表紙は採番しない。
- **箇条書きのインデント既定＝テキスト前 0.45cm／ぶら下げ 0.4cm**。PptxGenJS は両者を同値にしか出せないため、生成後に slide XML を後処理して厳密値（`marL="162000" indent="-144000"`）へ置換する（snippet は layout-patterns.md）。
- 生成後は **必ず PDF に変換して目視確認**（`soffice --headless --convert-to pdf`）。テーブル溢れ・タイトル折返し・余白を見る。
- **図解は原則 PptxGenJS のネイティブ図形（roundRect/line/text）で作る** — 納品先で編集できることを優先する。中程度までの複雑さ（構成図・責任分界・A/B案など）はこれで十分きれい。込み入った図は **drawio で設計・提案 → 同じ座標・配色をネイティブ図形へ移植して再現**（PNG埋め込みはしない）。詳細は layout-patterns.md 末尾。
- 直接編集された pptx を受け取ったら、**スライド順・図形座標・テキストを抽出して差分**を取り、生成スクリプトへ反映できる（生成スクリプトが正本）。

詳細な運用Tips（フォントフォールバック、タイトル1行化、テーブル行数の上限、図の扱い、直接編集の取り込み）は layout-patterns.md 末尾を参照。

---

## 重要な運用ルール

1. **STEP 2 の構成表ユーザー確認を省略しない。**
2. **原則 > テンプレ。** レイアウトはメッセージに合わせて裁量で選ぶが、principles.md の原則は外さない。
3. **色はブランドに合わせる**（theme で差し替え）。レイアウトの自由度はその上で確保する。
4. **事実には必ず So-What を添える**が、ラベルとして「示唆」の語は使わない（`▶` などのマーカーで示す）。
5. **会議・ヒアリングの温度感を正確に反映**し、相手が消極的な領域を過剰提案しない。確度に応じて「（案）」「（仮説）」を付す。
6. **便益は可能なら定量**（仮値なら「(仮)」明示）。
7. 生成物は **PDFで必ず目視確認**してから納品。
8. このスキルは汎用。**ブランド専用スキルがある案件はそちらを優先**。
